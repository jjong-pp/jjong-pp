# 🛠️ 다중 도메인 처리: B2C 솔루션으로 B2B를 구현하다
이전 글에서 기획했던 '결제 수수료 0% 방어 아키텍처'를 빠른 시간 내에 구현하기 위해, 프론트엔드를 바닥부터 개발하는 대신 기성 쇼핑몰 솔루션인 **'고도몰(Godo Mall)'**을 커스터마이징하는 전략을 채택했습니다. 개발 공수(MM) 단축과 인프라 안정성을 확보하기 위한 결단이었습니다.

하지만 치명적인 문제가 있었습니다. 고도몰은 철저하게 **B2C(기업-소비자) 전용 솔루션**으로 설계되어 있다는 점입니다. 140개 조리원(B2B)마다 단가가 다르고, 여신 한도가 다르며, 월별 통합 세금계산서 역발행 기능 같은 것은 고도몰에 존재하지 않았습니다.

## 🔀 다중 도메인 라우팅 (Multi-Domain Routing)과 웹훅
고도몰의 태생적 한계를 극복하기 위해, 저는 고도몰을 철저히 '장바구니와 UI 껍데기'로만 취급하고, 모든 핵심 비즈니스 로직(결제, 정산, 단가)은 자체 구축한 백엔드(ERP)에서 통제하도록 아키텍처를 분리했습니다.

\`[아키텍처: 고도몰 웹훅 연동 및 다중 도메인 라우팅 구조도 - NotebookLM Godo Mall 문서 참조]\`

1. **단가 분기 (Multi-Domain Routing)**:
	A 조리원과 B 조리원의 상품 매입 단가가 다릅니다. 고도몰 기본 기능으로는 이 복잡한 단가 체계를 감당할 수 없어, 로그인 시 백엔드에서 해당 조리원의 고유 도메인(세션)을 판별한 뒤, ERP에 저장된 맞춤형 단가표를 실시간 API로 주입(Override)하는 라우팅 로직을 구현했습니다.
2. **웹훅(Webhook) 기반 비동기 여신 결제**:
	원장님이 장바구니에 담고 '결제'를 누르는 순간, 고도몰 내부의 PG 모듈 대신 저희 백엔드 서버로 결제 생성 웹훅(Webhook) API가 발송됩니다.

\`[코드스니펫: 고도몰 결제 웹훅 수신 및 여신 한도 차감 트랜잭션 (로컬 프로젝트 소스코드 발췌)]\`

## 🧾 한 치의 오차도 없는 정산 파이프라인
백엔드 서버는 고도몰에서 웹훅을 수신하자마자 다음의 작업을 단일 트랜잭션으로 처리합니다.

1. 조리원의 남은 **여신 한도**를 안전하게 조회 및 차감 (동시성 제어를 위해 DB Lock 또는 Redis 활용).
2. 한도가 정상 차감되면 WMS(창고시스템)로 즉시 출고 지시(SO) 데이터 전송.
3. 승인 완료된 결제 데이터를 **월정산 스케줄러(Cron Job) 대기열**에 적재.

매월 말일 자정이 되면, 정산 스케줄러가 한 달 동안 흩어져 있던 발주 데이터를 조리원(사업자번호)별로 Grouping하여 하나의 청구 덩어리로 만듭니다. 그리고 외부 세금계산서 연동 API를 호출하여 원장님들의 이메일로 정확히 역발행을 수행합니다.

---
> **Series Wrap-up**
> "B2C 솔루션으로는 B2B 모델을 구현할 수 없습니다"라는 편견을 뚫고, B2C 프론트엔드 뒤에 B2B 백엔드 코어를 숨겨 연동하는 우회(Bypass) 아키텍처를 성공시켰습니다. 주어진 도구(고도몰)의 한계에 굴복하지 않고, 웹훅과 다중 도메인 API 라우팅 기술로 한계를 뛰어넘어 **현장 업무의 83% 단축**을 이뤄낸 잊지 못할 아키텍처 챌린지였습니다.

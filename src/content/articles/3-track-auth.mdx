# 🔒 국가 기관 API 연동의 핵심: 무결성과 보안
지난 글에서 다뤘던 5개사 시스템 기획이 '소통과 비즈니스'의 영역이었다면, 이번 글은 철저한 **'엔지니어링'**의 영역입니다. 
관세청에 통관 데이터를 전송한다는 것은, 일반적인 외부 API를 호출하는 것과는 차원이 다른 보안과 데이터 정합성을 요구합니다. 만약 중간에 누군가 개인통관고유부호를 가로채거나, 전송 과정에서 네트워크 오류로 통관 요청이 누락된다면 이는 단순한 시스템 장애를 넘어 치명적인 서비스 신뢰도 하락으로 이어질 수 있습니다. 
이를 방어하기 위해 제가 설계한 **3-Track 인증 아키텍처**와 **안전한 트랜잭션 파이프라인**을 소개합니다.

## 🛡️ 3-Track 인증 아키텍처 설계
단순한 API Key 방식으로는 부족했습니다. 저는 보안 수준을 극대화하기 위해 3중 안전장치를 두었습니다.
\`[이미지: 3-Track 인증 시퀀스 다이어그램 - 로컬 아키텍처 다이어그램 참조]\`

1. **IP Whitelisting (네트워크 계층)**: 관세청 시스템과 직접 통신하는 미들웨어 서버의 고정 IP를 사전 등록하여, 인가되지 않은 외부 접근을 원천 차단했습니다.
2. **OAuth 2.0 (토큰 인증 계층)**: API 호출 시마다 동적으로 발급되는 만료 기한이 짧은 Access Token을 사용하도록 구현했습니다.
3. **Payload 암호화 (데이터 계층)**: 통신 구간을 HTTPS로 보호하는 것에 더해, 전송되는 JSON Payload 내의 민감 정보(주민번호 대체 부호 등)를 AES 기반으로 암호화 처리했습니다.

## 🔄 절대 누락되지 않는 API: 트랜잭션과 보상 로직
네트워크는 언제든 끊길 수 있습니다. 특히 트래픽이 몰리는 시스템은 간혹 타임아웃(Timeout) 에러를 뱉어내기도 합니다. 데이터가 전송되다가 끊어졌을 때, 해당 건이 누락되지 않도록 설계하는 것이 백엔드 엔지니어의 핵심 역량입니다.
\`[코드스니펫: API 타임아웃, 재시도(Retry), DLQ 처리 로직 - 로컬 소스코드 발췌]\`

저는 통신 상태값을 세분화했습니다. (Ready → Processing → Completed / Failed).
만약 API 호출이 타임아웃 되거나 실패(Failed) 상태로 떨어지면, 큐(Queue)를 활용해 **지수 백오프(Exponential Backoff)** 재시도 로직을 실행했습니다. 지정된 횟수를 초과하여 최종 실패한 데이터는 관리자 대시보드(Dead Letter Queue)에 알람과 함께 적재되어, 수동 재처리가 가능하도록 **안전망**을 구현했습니다.

> **Series Wrap-up**
> 외부 시스템과의 연동은 언제나 내가 통제할 수 없는 '불확실성'과의 싸움입니다. 관세청 API 연동 프로젝트를 통해 단순히 가이드라인 문서를 읽고 코딩하는 것을 넘어, **비즈니스 리스크를 어떻게 견고한 아키텍처로 방어할 것인가**에 대한 시야를 한 단계 넓힐 수 있었습니다.

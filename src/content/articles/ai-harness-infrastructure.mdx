# 🤖 무너지지 않는 AI: 회복 탄력성을 갖춘 Harness 인프라 구축기
> "AI는 반드시 환각(Hallucination)을 겪고, API는 반드시 끊어집니다."

최근 AI 에이전트를 실무에 도입하려는 시도가 폭발적으로 늘고 있습니다. 하지만 데모 환경에서 완벽하게 작동하던 챗봇이나 에이전트가, 실제 운영(Production) 환경에 배포되는 순간 처참하게 무너지는 것을 수없이 목격했습니다. 

LLM(Large Language Model)은 본질적으로 확률적(Probabilistic) 모델이기에 100%의 일관성을 보장하지 못합니다. 또한 외부 도구(Tool)를 호출하는 에이전트는 네트워크 지연이나 타임아웃의 위험을 항상 안고 있습니다. 
저는 이 '불확실성'을 시스템적으로 제어하기 위해, 에이전트가 예측 불가능한 짓을 하더라도 시스템 전체가 다운되지 않는 **회복 탄력적인 Harness(안전망) 인프라**를 설계했습니다.

## 🪢 Harness 아키텍처: 에이전트의 생명줄
Harness 아키텍처의 철학은 단 하나입니다. **"AI가 실패할 것을 전제로 시스템을 설계한다."**

\`[이미지: AI Harness 인프라 장애 조치(Failover) 흐름도 - NotebookLM AI 엔지니어링 노트 참조]\`

1. **상태 관리 및 체크포인트 (State Management)**: 
	단일 프롬프트에 의존하는 것은 매우 위험합니다. 저는 에이전트의 작업 파이프라인을 상태 머신(State Machine) 구조로 쪼개어 관리했습니다. 특정 단계에서 툴 호출이 실패하더라도 처음부터 다시 토큰을 태우며 시작하는 것이 아니라, 저장된 체크포인트(Checkpoint)부터 이어서(Resume) 실행할 수 있도록 설계했습니다.
2. **서킷 브레이커 (Circuit Breaker) 패턴**: 
	에이전트가 연동된 외부 API가 먹통이 되었을 때, LLM이 무한 재시도 루프에 빠져 엄청난 과금 폭탄(Token Cost)을 유발하는 것을 막아야 합니다. API 호출 연속 3회 실패 시 해당 도구를 차단(Trip)하고, 캐시 데이터를 반환하거나 Fallback 프롬프트로 우회하도록 백엔드를 방어했습니다.

## 🛡️ Self-Correction: 스스로 궤도를 수정하는 AI
AI 에이전트 개발에서 가장 까다로운 점은 "에러가 JSON 형태로 예쁘게 떨어지지 않는다"는 것입니다. LLM이 환각에 빠져 엉뚱한 텍스트 포맷을 반환하면 시스템 파서(Parser)가 곧바로 패닉을 일으킵니다.

저는 이를 방어하기 위해 **Self-Correction(자기 교정) 루프**를 Harness 레이어에 심었습니다. 시스템에서 파싱 에러가 발생하면 Exception을 던지고 죽는 대신, 에러 로그 자체를 다시 LLM에게 먹여주어 "네가 준 답변 포맷이 틀렸으니 JSON 스키마에 맞게 다시 반환해"라고 자동 재요청(Auto-Retry)하도록 만들었습니다.

이러한 인프라 레벨의 견고한 방어 기재 덕분에, 실 서비스 환경에서의 에이전트 Crash 비율을 99% 이상 예방할 수 있었습니다.

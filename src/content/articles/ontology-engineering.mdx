공급망 관리(SCM) 현업에서 일하는 개발자나 데이터 분석가라면 누구나 이런 경험이 있을 것입니다.
종합 ERP 시스템에는 PART-A789라는 코드로, 창고관리시스템(WMS)에는 SKU-9511이라는 코드로
협력사 시스템에는 또 다른 이름으로 등록된 **'똑같은'** 부품.
이들을 통합 분석하기 위해 우리는 오늘도 엑셀 VLOOKUP과 수작업 매핑 테이블에 의존하며 힘겨운 싸움을 하고 있습니다.
이 방식은 비효율적일 뿐만 아니라, 데이터가 추가되거나 변경될 때마다 깨지기 쉬운 **아주 위험한 구조**입니다.
만약 시스템이 PART-A789와 SKU-9511이 **'의미적으로 동일한 대상'**임을 스스로 이해할 수 있다면 어떨까요?
이 질문에 대한 해답이 바로 **온톨로지 공학**과 **지식 그래프**에 있습니다.
지금부터 이 기술들이 어떻게 데이터의 단순 나열을 넘어
**컴퓨터가 '이해'하고 '추론'하는 지식의 네트워크로 바꾸어 놓는지**
그 핵심 구성 요소인 **RDF, RDFS, OWL**을 중심으로 차근차근 알아보겠습니다.
---
## 1. RDF로 모든 것을 '주어-서술어-목적어'로 표현하기
온톨로지의 가장 기초가 되는 벽돌은 **RDF(Resource Description Framework)**입니다.
RDF의 핵심 아이디어는 세상의 모든 지식을 아주 단순한 **트리플(Triple)**
즉 **주어(Subject)** **- 서술어(Predicate) - 목적어(Object)**라는 세 요소의 조합으로 표현하는 것입니다.
마치 우리가 "A 창고는 - 서울에 - 위치한다"라고 말하는 것과 같습니다.
이 단순한 구조는 컴퓨터가 정보를 명확하게 처리할 수 있는 최소 단위가 됩니다.
여기서 가장 중요한 점은, 각 요소가 단순한 문자열이 아니라 **URI(Uniform Resource Identifier)**라는 고유한 웹 주소 형태의 식별자를 갖는다는 것입니다.
- 'A창고' (X) -\> \<http://mySCM.com/warehouse/A\> (O)
- '위치한다' (X) -\> \<http://mySCM.com/property/isLocatedIn\> (O)
- '서울' (X) -\> \<http://mySCM.com/city/Seoul\> (O)
이렇게 모든 개념에 고유 ID를 부여함으로써, '서울'이라는 이름의 도시와 '서울'이라는 이름의 사람을 컴퓨터가 절대 혼동하지 않게 됩니다. 즉, **데이터의 모호함이 원천적으로 제거**됩니다.
이러한 트리플들이 수없이 연결되면, 점과 선으로 이루어진 거대한 지식의 네트워크, 즉 **지식 그래프**가 만들어집니다.
#### 잠깐, 이게 관계형 데이터베이스(RDB)랑 뭐가 다른가요?
RDB가 정해진 틀(테이블 스키마)에 데이터를 맞추는 방식이라면
RDF는 데이터 조각(트리플)들을 레고처럼 자유롭게 연결하는 방식입니다.
<table>
<tr>
<td>구분</td>
<td>**관계형 데이터베이스 (RDB)**</td>
<td>**그래프 데이터 (RDF)**</td>
</tr>
<tr>
<td>**구조**</td>
<td>정해진 스키마 (테이블, 행, 열)</td>
<td>스키마 없는 유연한 구조 (트리플)</td>
</tr>
<tr>
<td>**관계 표현**</td>
<td>외래 키(Foreign Key)를 이용한 **JOIN** 연산</td>
<td>관계로 노드를 직접 연결 (**Graph Traversal**)</td>
</tr>
<tr>
<td>**유연성**</td>
<td>스키마 변경이 매우 복잡하고 비용이 큼</td>
<td>새로운 관계(Predicate)나 데이터 타입 추가가 자유로움</td>
</tr>
</table>
SCM 데이터처럼 공급사, 고객사, 물류센터 등 계속해서 관계와 종류가 변하는 복잡한 환경에서는 RDF의 유연성이 강력한 무기가 됩니다.

## 2. RDFS로 데이터에 '분류 체계(문법)' 부여하기
RDF만으로는 데이터의 의미를 충분히 표현할 수 없습니다. \<http://mySCM.com/warehouse/A\>가 '창고'라는 **개념**에 속하는지 아니면 '부품'인지 컴퓨터는 알지 못합니다.
이때 필요한 것이 바로 **RDFS(RDF Schema)**입니다.
RDFS는 우리가 만든 RDF 데이터에 대한 **'기본적인 문법 규칙'과 '분류 체계**'를 정의합니다.
RDFS는 몇 가지 중요한 용어를 제공하여 데이터에 질서를 부여합니다.
- **rdfs:Class**: '개념' 또는 '분류'를 정의합니다. (예: 창고, 제품, 공급업체)
- **rdfs:subClassOf**: 특정 클래스가 다른 클래스의 하위 분류임을 명시합니다. (예: 냉장창고는 창고의 하위 분류)
- **rdfs:domain**: 특정 관계(서술어)의 **주어**가 될 수 있는 클래스를 제한합니다.
- **rdfs:range**: 특정 관계(서술어)의 **목적어**가 될 수 있는 클래스를 제한합니다.
#### RDFS가 왜 중요한가요?
"오직 **창고**만이 **제품**을 보관할 수 있다"는 규칙을 시스템에 알려줄 수 있습니다.
```plain text
:stores a rdf:Property ;
         # 주어(domain)는 반드시 '창고' 클래스여야 함
         rdfs:domain :Warehouse ;
         # 목적어(range)는 반드시 '제품' 클래스여야 함
         rdfs:range :Product .
```
만약 누군가 '제품이 창고를 보관한다'와 같은 논리적으로 말이 안 되는 데이터를 입력하면 시스템은 이 규칙에 어긋남을 스스로 감지할 수 있게 됩니다.
이처럼 RDFS는 데이터의 **정합성과 품질을 보장**하는 최소한의 안전장치 역할을 합니다.

## 3. 세 번째 벽돌: OWL로 '논리적 추론 시스템' 구축하기
RDFS는 기본적인 분류 체계는 제공하지만, 더 복잡한 현실 세계의 비즈니스 규칙을 표현하기에는 한계가 있습니다.
- "모든 부품은 **단 하나의** 고유 부품 번호만 가질 수 있다."
- "A 부품의 공급사는 B사이고, B사의 협력사는 C사라면, C사도 A 부품의 **잠재적 공급망**에 속한다."
- "ERP의 PART-A789와 WMS의 SKU-9511은 **사실 동일한 부품**이다."
이러한 정교하고 복잡한 논리를 표현하고 이를 바탕으로 시스템이 새로운 사실을 **'추론'**하게 만드는 강력한 언어가 바로 **OWL(Web Ontology Language)**입니다.
OWL은 RDFS를 확장하여 컴퓨터가 마치 사람처럼 논리적 사고를 할 수 있는 강력한 기능을 제공합니다.
#### OWL의 추론 능력
OWL이 제공하는 몇 가지 핵심 기능만 봐도 그 강력함을 느낄 수 있습니다.
<table>
<tr>
<td>OWL 문법</td>
<td>의미(논리 규칙)</td>
<td>SCM 활용 예시</td>
<td>**시스템의 추론 결과**</td>
</tr>
<tr>
<td>**sameAs**</td>
<td>**동일성 선언**</td>
<td>:PART-A789 owl:sameAs :SKU-9511 .</td>
<td>:PART-A789를 검색하면 :SKU-9511의 정보까지 **자동으로 함께 조회**됩니다. (서두의 문제 해결!)</td>
</tr>
<tr>
<td>**TransitiveProperty**</td>
<td>**추이적 관계**</td>
<td>:isLocatedIn(…에 위치한다)을 추이적 속성으로 정의. (부천공장은 부천에 위치, 부천은 경기도에 위치)</td>
<td>우리는 명시하지 않았지만, 시스템은 **부천공장이 경기도에 위치한다는 새로운 사실을 스스로 추론**해냅니다.</td>
</tr>
<tr>
<td>**FunctionalProperty**</td>
<td>**유일성 보장**</td>
<td>:hasPartNumber를 함수적 속성으로 정의.</td>
<td>한 부품에 두 개 이상의 부품 번호를 할당하려 하면, 시스템이 **논리적 모순임을 감지**합니다.</td>
</tr>
<tr>
<td>**inverseOf**</td>
<td>**역관계 정의**</td>
<td>:isSuppliedBy는 :supplies의 역관계.</td>
<td>A사가 B부품을 공급(supplies)한다고 입력하면, B부품은 A사에 의해 공급된다(isSuppliedBy)는 사실이 **자동으로 생성**됩니다.</td>
</tr>
</table>
이러한 규칙들을 바탕으로 새로운 사실을 추론해내는 소프트웨어 엔진을 **추론기**라고 부릅니다.
OWL과 추론기를 통해 우리의 SCM 데이터 시스템은 단순히 데이터를 저장하는 창고에서 새로운 지식을 스스로 생성하고 확장해나가는 지능적인 '**지식 베이스(Knowledge Base)'**로 진화합니다.

## 실전 활용: SPARQL로 지식 그래프에 질문하고 답 얻기
자, 이렇게 지식 그래프를 구축했다면 어떻게 활용할 수 있을까요?
지식 그래프에 질문을 던지는 표준 언어가 바로 **SPARQL(스파클)** 입니다.
SPARQL은 '**그래프를 위한 SQL'**이라고 생각하면 쉽습니다.
우리가 처음 제기했던 문제, 즉 ERP의 PART-A789와 관련된 모든 정보를 찾고 싶을 때 SPARQL을 사용하면 어떻게 될까요?
```plain text
# 'PART-A789'와 관련된 모든 이름과 재고 위치를 알려줘

SELECT ?name ?location
WHERE {
  # ?part는 PART-A789 이거나, 그와 동일한(sameAs) 모든 것을 의미
  { :PART-A789 ?p ?o } UNION { ?s ?p :PART-A789 } .
  BIND(IF(BOUND(?s), ?s, :PART-A789) AS ?part)

  # ?part와 관련된 이름과 위치 정보를 찾음
  ?part :hasName ?name .
  ?part :isStoredIn ?warehouse .
  ?warehouse :isLocatedIn ?location .
}
```
이 쿼리의 놀라운 점은 우리는 PART-A789만 물어봤지만 OWL의 owl:sameAs 규칙 덕분에
시스템은 알아서 SKU-9511과 관련된 정보(예: WMS 상의 창고 위치)까지 **모두 찾아서 한 번에** 보여준다는 것입니다.
더 이상 여러 시스템을 오가며 수작업으로 데이터를 취합할 필요가 없습니다.

## 결론: 데이터를 넘어 지식으로, 미래 SCM의 핵심 인프라
RDF, RDFS, OWL. 언뜻 복잡해 보이지만 이 기술들의 여정은 결국 한 가지 목표를 향합니다.
바로 **'지식을 정확하고 명료하게 표현하여 기계가 이해하고 활용하게 만들자'**는 것입니다.
- **RDF**는 지식의 최소 단위(알파벳)를 정의하고
- **RDFS**는 그 지식들의 기본 분류 체계(기본 문법)를 세우며
- **OWL**은 그 지식들 사이에 흐르는 논리(고급 문법)를 부여하여 추론을 가능하게 합니다.
이러한 정교한 설계도를 통해 우리는 데이터의 모호함을 제거하고 여러 시스템에 흩어진 데이터를 완벽하게 통합하며,
궁극적으로는 시스템이 데이터의 **'의미'**를 이해하여 사람의 의사결정을 돕는 진정한 AI 시스템을 구축할 수 있습니다.
특히 환각(Hallucination) 현상이 문제 되는 최신 LLM(거대 언어 모델) 시대에
사실에 기반한 **지식 그래프는 AI가 신뢰할 수 있는 답변을 생성하도록 돕는 핵심적인 역할**을 합니다.
